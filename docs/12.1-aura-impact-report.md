# DandersFrames — Patch 12.1 Aura Lockdown: Impact Report

**Date:** 2026-06-19
**Researched against:** `Gethe/wow-ui-source` tag **`12.1.0`** (PTR build **68209**) — generated API docs, the new `Blizzard_AuraContainer` reference addon, and Blizzard's consumer frames. Cross-checked vs the 12.1 dev-notes.

## Planning basis (read this first)

This report is written for the **end-state Blizzard has signalled**, not the partial enforcement in today's PTR build:

1. **Full UnitAura lockdown.** `GetUnitAuras` / `GetUnitAuraInstanceIDs` become **secret vectors** — addon code cannot get a count, iterate them, or pull per-aura handles out. All UnitAura APIs return secrets or nil.
2. **No meaningful whitelist.** Assume essentially **all auras become secret again** — the current readable set (HoTs, defensives, class buffs that `AuraAdapter` reads by spellId) goes away. Any residual flagged-readable spells are a *bonus, not a foundation*.
3. **Display happens through `AuraContainer`/`AuraButton`**, which Blizzard fills internally in the secure environment; addons style, they do not identify.

Build 68209 is a **partial waypoint** toward this (see Part 4) — useful for prototyping, but we must design for #1–#3.

**Tags:** **[end-state]** = the planning basis above · **[68209]** = verified in current PTR source · **[DF]** = our code · **[open]** = unresolved, decides the outcome.

## TL;DR

- **[end-state]** Under full lockdown with no whitelist, **DF cannot identify auras in its own code.** No spellId (all secret), no enumeration (secret vectors), and **fingerprinting collapses** (see below). The clever `SecretAuras.lua` technique and the `AuraAdapter.lua` spellId track both stop being foundations.
- **[DF] Why fingerprinting dies:** it is a *small-set disambiguator*, not a general identifier. It only runs on the residual secret auras *after the whitelist filters everything else out* (`MatchAuraSignature` literally `return nil` for non-secret auras), and it distinguishes them with a **4-bit filter signature** (16 values) plus hand-written per-spec collision engines. Remove the whitelist and the *entire* aura pool floods that 16-value space — signatures collide hopelessly. It cannot identify arbitrary spells, and most current identifications break.
- **[end-state] What this means for AD:** Aura Designer stops being "identify any aura, render it where you want." It splits into:
  - ✅ **Category displays** (via `AuraContainer`): "my HoTs," "my debuffs," "dispellable debuffs," "big defensives" — styled by us, filled by Blizzard, but **we don't know which box is which aura**.
  - ⚠️ **Self-cast indicators** (aura-independent): track *your own* casts via `UNIT_SPELLCAST_SUCCEEDED` + a self-run timer — never touches UnitAura. Survives, but bespoke, fragile on target attribution, and blind to refresh/dispel.
  - 🔴 **Specific-spell placement of anything you didn't cast** (e.g. "show *this other healer's* Rejuv as a box here," or per-spell styling within a category): **depends entirely on whether `AuraContainer` gains per-spell custom filters** — which it does **not** have today and Blizzard has not shown.
- **[open] The single decisive question:** **Will `AuraContainer.AddAuraFilter` support per-spell filtering (spellID/name)?** If yes, AD's targeted indicators largely survive (Blizzard does the identifying inside the secure container). If no, AD targeted indicators are limited to category boxes + self-cast timers — a large capability loss.
- **Unaffected:** Targeted Spells (no UnitAura), Private Auras (sanctioned anchors).

## Part 1 — The lockdown model we're planning for

### 1.1 What "fully locked down" means for addon code

**[end-state]** Insecure addon code (i.e. all of DF outside a secure container) gets, for any restricted unit/aura:
- **No spellId** — every aura's `spellId` is a secret value; can't read it, can't compare it to a known number.
- **No enumeration** — `GetUnitAuras`/`GetUnitAuraInstanceIDs` are secret vectors; no `#`, no `ipairs`.
- **No per-aura data** — `GetAuraDataByAuraInstanceID/ByIndex/BySlot`, duration, stacks, dispel type, expiration all return secret values (display-only; no Lua branching).
- **By-spell lookup blocked** — `GetUnitAuraBySpellID`/`GetAuraDataBySpellName` carry `RequiresNonSecretAura` → nil for secret (i.e. all) spells.

### 1.2 The defining variable: do insecure aura **instance IDs** survive?

Everything hinges on whether *any* non-secret `auraInstanceID` still reaches insecure code (via the `UNIT_AURA` event delta), because an instance ID is the only thing you could feed to the one classifier that is still non-secret today (`IsAuraFilteredOutByInstanceID`).

- **[68209] Today:** the `UNIT_AURA` `updateInfo.addedAuras`/`removedAuraInstanceIDs` delta still delivers instance IDs, and `IsAuraFilteredOutByInstanceID` is still a plain `bool`.
- **[end-state] Very likely:** if Blizzard left a non-secret instance-ID stream in the event, the whole secret-vector effort would be pointless (you'd rebuild your own list from the event). The intended model is **insecure code never touches aura identity** — the secure `AuraContainer` does. So plan for the delta's instance IDs to become **secret/unusable** to insecure code.

**Consequence:** in the end-state, `IsAuraFilteredOutByInstanceID` being non-secret is **moot** — you have no instance ID to pass it. The "linchpin" I previously emphasized only matters in the milder world where the event delta survives; do **not** build the plan on it.

### 1.3 What stays usable to insecure code (aura-wise): effectively nothing

No spellId, no enumeration, no instance IDs, no readable data. The only aura-adjacent things insecure code keeps are:
- **Your own cast events** (`UNIT_SPELLCAST_SUCCEEDED` spellId is your action, non-secret).
- **Private-aura anchors** (`Add/RemovePrivateAuraAnchor`).
- **`AuraContainer`** as a styling surface (Blizzard identifies + fills; you style).

## Part 2 — What survives, by mechanism

| Mechanism | Survives? | What it can do | What it cannot do |
|---|---|---|---|
| **Non-aura features** (Targeted Spells, range, health, etc.) | 🟢 **Yes, unchanged** | `UNIT_SPELLCAST_*`, `UnitHealth`, etc. — no UnitAura dependency. | n/a |
| **Private Auras** (`PrivateAuras.lua`) | 🟢 **Yes** | Sanctioned `Add/RemovePrivateAuraAnchor` — same path `CompactUnitFrame` uses. | Customising the aura visuals (Blizzard-drawn). |
| **Category `AuraContainer`** | 🟢 **Yes** | Show a fixed pool of styled boxes for a **filter category** — `HELPFUL`/`HARMFUL`/`PLAYER`/`RAID`(dispellable)/big-defensive (per `AuraFilters` + sort rules). We supply icon/cooldown/border regions; Blizzard fills + drives them. | **Know which box holds which aura.** No per-spell selection, no per-aura styling, no reading stacks/duration/IsShown (all secret). |
| **Self-cast aura-independent timer** | ⚠️ **Partial** | Track *your own* `UNIT_SPELLCAST_SUCCEEDED` and run your **own** duration timer + indicator at any position — never queries UnitAura, so lockdown-immune. | Others' auras; accurate refresh/dispel/early-loss (no aura feedback); reliable target attribution (cast event lacks target — must be inferred). |
| **Spell-specific `AuraContainer`** | 🔴 **[open]** | *Only if* `AddAuraFilter` gains a spellID/name option — then Blizzard filters to that spell inside the secure container and fills one button you place. | Exists only if Blizzard ships per-spell filters. **Not in 68209.** |
| **Addon-side identification** (`SecretAuras` fingerprint, `AuraAdapter` spellId) | 🔴 **No** | — | Collapses without the whitelist (fingerprint) / readable spellIds (adapter). A residual handful of auras with a *globally* unique filter signature might still resolve, but it is not a foundation. |
| **Conditional logic** (stacks ≥ N, duration < X, combos) | 🔴 **No (under restriction)** | — | All inputs secret; can't branch in Lua. Some thresholds can be pushed into Blizzard-driven facilities (`textColorCurve`, cooldown state) inside a container. |

## Part 3 — Aura Designer, per indicator type (end-state basis)

**[DF]** AD has **9 indicator types** in two families. Two cross-cutting truths reshape all of them under the end-state:
- **Placement was never the problem** — it's our own `SetPoint`. The problem is **identification**: we can no longer know *which* aura is present, except for self-casts (cast events) or via a category/spell container (Blizzard identifies).
- So each type's fate = "can it be driven by a source that survives?" — i.e. an `AuraContainer` category fill, a per-spell container (if filters ship), or a self-cast timer.

**Family A — placeable indicators (`icon`, `square`, `bar`):**

| Capability | End-state verdict | Notes |
|---|---|---|
| Arbitrary placement | 🟢 Works | Our frame, our `SetPoint`. Never in question. |
| Show a **category** (my HoTs / my debuffs / dispellable) as boxes | 🟢 Works via container | Styled by us, filled by Blizzard; can't label which is which. |
| Show **one specific spell** (others' Rejuv) at a position | 🔴 **[open]** | Needs per-spell container filter (not shipped) — else impossible. |
| Show **your own** spell at a position | ⚠️ Self-cast timer | Aura-independent; caveats (target attribution, no refresh feedback). |
| Icon texture / cooldown ring / stack text | 🟠 Display-only | Inside a container these are Blizzard-driven (secret pass-through); no Lua reads. |
| Per-aura distinct styling (Rejuv green, Regrowth blue) | 🔴 No | Can't tell buttons apart in a category container. |

**Family B — unit-frame modifiers (`border`, `healthbar`, `background`, `nametext`, `healthtext`, `framealpha`):**

| Capability | End-state verdict | Notes |
|---|---|---|
| Recolour/alpha the frame **on presence of a category** (e.g. "any dispellable debuff → red border") | 🟢/🟠 Works via container-driven trigger | A category container can drive a presence signal; the recolour is our config colour. Dispel-type colour comes through secret → `SetColorTexture` only (not `SetGradient`). |
| Recolour on **a specific non-cast spell** | 🔴 **[open]** | Same per-spell-filter dependency. |
| Recolour on **your own** spell | ⚠️ Self-cast timer | As above. |
| `border` DF Dash expiring animation timing | 🟠 Rework | Drive a registered `Cooldown`; no Lua "time-remaining" math (secret). |

**Cross-cutting AD behaviours:**

| Behaviour | End-state verdict |
|---|---|
| "Show when **missing**" (absence) | 🔴 Mostly lost — needs identity tracking we no longer have for others' auras; possible only for self-cast (you know you didn't cast / it expired on your timer). |
| "Show when **stacks ≥ N**" | 🔴 Lost (count secret). |
| "Flash when **duration < X**" | 🔴 Lost in Lua; 🟠 only via Blizzard `textColorCurve` inside a container. |

### The honest answer to "can we put Rejuv exactly where we want?"

- **Your own Rejuv:** ⚠️ **yes, but rebuilt** as a self-cast, aura-independent indicator (cast event + self timer), with the caveats above — not via reading the aura.
- **Anyone else's Rejuv, or per-spell styling:** 🔴 **only if `AuraContainer` ships per-spell custom filters.** That is the unanswered question, and under the end-state it is **the** determinant. Without it, the most we can do is a category box ("a HoT is here") that we can't identify as Rejuv specifically.

## Part 4 — Current PTR build (68209): verified facts

These are confirmed in source and remain the substrate (enforcement is *partial* today — see staging):

- **Secrecy model:** flag `SecretWhenUnitAuraRestricted` → returns become **secret values** (not nil) under combat/encounter/M+/PvP. Per-spell secrecy queryable via **`C_Secrets.GetSpellAuraSecrecy(spellID) → SecrecyLevel`** (← use this to measure the real whitelist on PTR).
- **`AuraContainer`/`AuraButton`** (`Blizzard_AuraContainer`, `## UseSecureEnvironment: 1`): Lua intrinsics; usable surface = `CustomAuraContainerTemplate`/`CustomAuraButtonTemplate`.
  - **`AddAuraFilter(filterString, {maxFrameCount, roundUpFrameIndex})`** — filter is the classic `AuraFilters` flag string; **no spellID/name/caster option exists.** Assignment is `AuraUtil.DefaultAuraCompare` order, capped by `maxFrameCount`.
  - `AuraButton` is a **slot registry**: you register your own `Icon`/`Cooldown`/`FontString`/`StatusBar`; Blizzard pushes `secretwrap`-ed data in. `SetShown(secretwrap(...))` → `IsShown` is secret. Decorative border child-textures attach freely.
- **Forbidden Aspects:** only **two** shipped — `SetToDefaults`, `ScriptBindings`. The blog's `UntrustedScriptExecution` **did not ship**. Aura objects use the **public/forbidden partition split** (`inboundPartition="public"`), not aspects.
- **Roleset System:** `C_Roleset.ApplyRolesetFilters(blocked, allowed)`; membership via XML `roleset="…"`; force-hides regardless of shown state. `"buffs"`/`"unitFrames"` are first-class blocklist rolesets.
- **Misc:** `Frame:SetOnUpdateMode` (Disabled/RunWhenVisible/RunWhenVisibleOnce/RunOnce/RunAlways); `VectorGraphics`/SVG (`SetSVG`…); radial mask on TextureBase (`SetRadialProgressBarPercent`…); `[Bootstrap]` TOC directive (+ `UIParent.lua` refactor); `C_InterfaceFileManifest.GetInterfaceArtFiles()` (new texture names no longer published).

**Staging (why 68209 ≠ end-state):** Blizzard's raid/party `CompactUnitFrame` is **fully migrated** off Lua aura iteration (drives a native container via secure attributes), but their **own player `BuffFrame` still calls `AuraUtil.ForEachAura`** + index tooltips — proof full enforcement isn't live yet. `GetUnitAuraInstanceIDs`/`GetAuraSlots` not yet flagged secret; `GetUnitAuras` returns an iterable (contents-secret) table. So today addon iteration largely still works; it tightens "over the following weeks," converging on Part 1.

## Part 5 — Decisive questions & PTR tests

| Question | How to settle it on PTR | Why it matters |
|---|---|---|
| **Does `AddAuraFilter` support per-spell filters?** | Read the final-build `Blizzard_AuraContainer` source; test `AddAuraFilter` with a spell option. | **THE** determinant of whether AD's specific-spell indicators survive. |
| How secret are *our* tracked spells really? | `C_Secrets.GetSpellAuraSecrecy(774)` (Rejuv) etc. across our spell list. | Measures the real whitelist — confirms/denies "all secret." |
| Do insecure `auraInstanceID`s survive in the `UNIT_AURA` delta? | On a restricted unit, check whether `updateInfo.addedAuras[i].auraInstanceID` is `issecretvalue`. | Decides whether *any* addon-side identification (fingerprint/cast-tag) remains possible. |
| Can a self-cast timer get its **target**? | Inspect cast events / `UnitSpellTargetName`-style APIs for restricted casts. | Decides whether self-cast indicators can be unit-placed. |
| Can an **addon-authored** secure container mixin enumerate auras? | Prototype a DF mixin on the `AuraContainer` intrinsic; test enumeration under restriction. | Only relevant if we attempt our own identification inside the secure env. |

## Part 6 — Recommended sequencing

1. **Measure the whitelist** with `C_Secrets.GetSpellAuraSecrecy` for DF's tracked spells — confirm the "all secret" basis empirically (a short `/df` command, not a giant `/run`).
2. **Settle the per-spell-filter question** from final-build container source — this gates the whole AD redesign.
3. **Prototype the two survival paths:** (a) a category `AuraContainer` with DF-styled buttons; (b) a self-cast aura-independent indicator for the healer's own HoTs.
4. **Reposition AD's value prop** around what survives: category boxes + self-cast indicators + (if filters ship) per-spell containers — and be candid in the UI about what can no longer be tracked under restriction.
5. Keep Targeted Spells / Private Auras as-is.
